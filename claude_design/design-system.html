01 / reframing

Component Library ではなく Design System Platform

個々のコンポーネントを並べただけでは "倉庫" にしかなりません。 AniService のフロントエンドを 長期再利用可能な基盤 にするため、 Design System を 6つの層 に分けて設計します。

fig.1 — Design System の6層
L6
Apps
EniTrance / EniDB / ServiceA / ServiceB の実プロダクト
L5
Patterns
LoginForm / TenantSwitcher / PermissionEditor / DataTable など、ドメイン固有の組み合わせ
L4
Components
Button / Input / Dialog / Select / Card — shadcn/ui 相当の汎用部品
L3
Primitives
Radix Primitives — 見た目ゼロの a11y 部品（自分では書かない）
L2
Tokens
Colors / Spacing / Typography / Motion / Radii — デザインの単一の真実
L1
Foundations
Tailwind CSS 4 / TypeScript strict / Biome / Turborepo

※ 階層の意義: 下層は変えにくく、上層は変わりやすい。L2 Tokens を変えると全層に波及、 L5 Patterns を変えても L4 以下は無傷。これが design system の本質です。

02 / repository

Monorepo — packages × apps

Turborepo + pnpm workspace で 1リポジトリにすべてを統合。 packages/ が再利用資産、apps/ が消費者。変更は即座に全アプリに反映され、 npm publish のサイクルを待つ必要がありません。 apps/storybook と apps/docs は用途が異なるため分離し、 @eni/config は tsconfig / tailwind / biome をサブディレクトリで管理します。

04 / differentiators

差別化ポイント — 他案では出てこない10項目

"コンポーネントを集めました" で止まらない、プラットフォーム視点 でしか出てこない設計判断。 これがコンペで効きます。

01

Single Source of Truth な Token 設計

@eni/tokens がすべての色・サイズ・タイポの唯一の真実。 Style Dictionary で Figma Variables にも export。 デザイナーが Figma で色を変えるとコードに流れる仕組み。

02

多テーマ・ホワイトラベル対応

CSS変数で配信するので <html data-tenant="customer-x"> で 色が切り替わる。将来 ServiceA を顧客ブランドで出すホワイトラベル提供にも対応可能。

03

GraphQL Codegen で EniDB と型同期

EniDB のスキーマから TypeScript の型と React hooks を自動生成。 useDealsQuery() を型安全に呼べる。 フロント/バックの型ズレ事故を構造的に防止。

04

CLI で新サービス scaffold

pnpm create @eni/app service-c で auth + ui + tokens 全部セットアップ済みの新規アプリが10分で生まれる。 "再利用可能" を実演する強い証拠。

05

ビジュアル回帰テスト

Chromatic で全コンポーネントのスクリーンショットを CI で比較。 PR 毎に "見た目が変わった箇所" がレビュアーに見える。 デザインシステム運用のプロ感。

06

アクセシビリティ CI

axe-core を Storybook と Playwright 両方に組み込み、 WCAG AA 違反を PR でブロック。 法務リスク・公共調達対応も視野に入っていることを示せる。

07

Performance Budget

bundle size の上限を CI で監視、LCP / INP を Vercel Analytics で計測。 "美しさだけじゃなく速度も" 設計済み。

08

ドキュメント自動生成

Storybook + MDX でコンポーネント props は TSDoc から自動抽出、 使用例は CodeSandbox 埋め込み。 "コードと文書がズレる" 問題を構造的に解消。

09

段階的移行ストーリー

Radix Themes の既存コードから新システムへの段階的移行プラン。 どのスプリントで何を置き換えるか明示。 机上の空論ではない証拠。

10

ADR と onboarding 設計

Architecture Decision Records を docs/adr/ に蓄積、 README が新規メンバーの onboarding entry。 個人プロジェクトではなく組織アセットとして運用。

07 / migration

既存リポへの導入 — 急がず壊さず

anitrance / anidb は独立リポとして存続し、@eni/ui を pnpm install して使う形になります。 既存の @radix-ui/themes ベースのコードを捨てる必要はなく、 新規 PR から段階的に置き換える戦略です。

step 1

共存期間

@radix-ui/themes と @eni/ui を同じアプリ内で共存。 新規画面は @eni/ui、既存画面はそのまま。

step 2

境界の置き換え

高頻度修正される画面から @eni/ui へ書き換え。 1 PR = 1画面のスコープで安全に進める。

step 3

完全置換

全画面の移行完了後、@radix-ui/themes を package.json から外す。 bundle size 削減効果も得られる。

並行運用が成立する理由

@radix-ui/themes の <Theme> プロバイダーは、 その内側の Themes コンポーネントだけに影響します。@eni/ui は Radix Primitives + Tailwind なので、Themes の CSS と干渉しません。 段階移行が技術的に成立する設計です。

08 / roi

投資対効果 — なぜいま作るか

Design System への投資は、サービス数が増えるほど指数関数的に効きます。 AniService が4サービス目を立ち上げるタイミングが分岐点。

従来の進め方 Design System Platform
新サービス立ち上げ時間 2-4週（ログイン / ベース UI / レイアウトを毎回作る） 1-2日（create @eni/app + 業務ロジックのみ）
ブランド変更コスト 全サービスで個別修正 tokens 1ファイル変更で全サービスに波及
a11y 改善コスト サービス毎にレビュー・修正 共通コンポーネント1箇所修正で全サービス改善
新メンバー onboarding サービス毎にコードベース把握 1リポジトリを理解すれば全サービスに参画可
セキュリティパッチ 各サービスで個別対応 共通パッケージ更新で全サービスに反映
デザイナー連携 Figma と実装が乖離 tokens の自動同期で常に一致

※ 初期投資は 16週 × 1人 程度（既存サービスの移行は別カウント）。 ServiceA / B 立ち上げ時に投資回収。3つ目のサービスから純粋な利益になります。

08 / private distribution

プライベート配布 — GCP IAM で守る

Design System は 社内専用。GCP Artifact Registry に publish し、 IAM 権限を持つ社員だけが pnpm install @eni/ui で利用できます。 EniTrance / EniDB と 同じセキュリティ境界 で運用できるのが最大の利点。

fig.3 — 配布と認可のフロー
@eni/ui
pnpm publish
① publish
GCP Artifact Registry
eni-platform/eni-npm
② pnpm install
社員のPC / CI
gcloud auth で認証

※ 権限がない人が叩くと npm error 403 Forbidden で install 自体が失敗。 退職者の artifactregistry.reader を IAM から剥がせば、その時点で install 不可になります。

配布側 — Artifact Registry を1回作る

# レジストリ作成（初回のみ）
gcloud artifacts repositories create eni-npm \
  --repository-format=npm \
  --location=asia-northeast1 \
  --project=eni-platform

# 設計者チームに writer 権限
gcloud artifacts repositories add-iam-policy-binding eni-npm \
  --location=asia-northeast1 \
  --member=group:[email protected] \
  --role=roles/artifactregistry.writer

# 全社員に reader 権限
gcloud artifacts repositories add-iam-policy-binding eni-npm \
  --location=asia-northeast1 \
  --member=group:[email protected] \
  --role=roles/artifactregistry.reader

利用側 — プロジェクト .npmrc 1ファイルだけ

# .npmrc
@eni:registry=https://asia-northeast1-npm.pkg.dev/eni-platform/eni-npm/

# 社員は1回だけ：
gcloud auth login
gcloud auth application-default login

# あとは普通に：
pnpm install @eni/ui

# これでもう使える
import { Button } from "@eni/ui";

IAM 権限マトリクス

ロール 付与対象 できること
artifactregistry.reader 全社員グループ pnpm install ができる
artifactregistry.writer 技術開発部 チームグループ 新しいバージョンを publish できる
artifactregistry.admin リード / あなた IAM 自体の変更・レジストリ設定
権限なし（外部・退職者） — install 自体が 403 で失敗
なぜ GitHub Packages ではなく Artifact Registry か
  • 既存の GCP IAM に乗る — EniTrance/EniDB と同じ権限管理。退職時の権限剥奪が1箇所で完結
  • Cloud Run からの認証が自動 — service-account をそのまま使える（PAT発行不要）
  • Cloud Audit Logs に集約 — 「誰がいつ install したか」が既存のログ基盤に流れる
  • VPC Service Controls 対応 — perimeter の中に閉じ込められる（GitHub Packages は外部）
  • コストが小さい — Org メンバー数で課金されない（パッケージ容量とリクエスト数のみ）

提案サマリ

AniService の Frontend を、単発のコンポーネント集ではなく、長期再利用可能なプラットフォームとして設計します。 Token / Component / Pattern / App の6層、12+ packages の monorepo、自動同期されるデザイントークン、 GraphQL codegen、ビジュアル回帰、a11y CI、CLI scaffold — すべて業界標準とアラインメント済み。 3サービス目を超えた瞬間に純粋な利益を生み、 将来のホワイトラベル提供にも構造的に対応できる基盤です。
